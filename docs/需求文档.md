### 任务看板

基于 next.js + prisma + SQLite 全栈：看板任务管理（KanBan Board）
· 目标：实现类似 Trello 的任务拖拽（Todo/Doing/Done）及数据同步。

1. 功能模块划分
   1.1 看板管理
   创建新看板
   编辑看板名称
   删除看板
   切换不同看板

1.2 列管理（待办/进行中/已完成）
添加新列
编辑列名称
删除列（级联删除列下所有任务）
拖拽调整列顺序
默认三列：Todo、Doing、Done

1.3 任务卡片管理
创建任务
任务标题（必填）
任务描述（可选）
标签选择（可选）
截止日期（可选）
编辑任务
删除任务
拖拽任务到不同列
拖拽调整同列内的任务顺序
任务详情弹窗

1.4 标签系统
创建标签
编辑标签（名称、颜色）
删除标签
为任务添加/移除标签

1.5 数据持久化
所有操作实时保存到数据库
页面刷新后数据不丢失
拖拽操作自动同步

1.6 性能优化
使用 React.memo 优化任务卡片渲染
虚拟滚动处理大量任务
API 请求防抖/节流
图片懒加载
使用 Next.js 的 App Router 实现服务端组件和客户端组件分离

2. 用户体验要求
   2.1 交互反馈
   拖拽时有半透明预览效果
   拖拽时显示可放置区域高亮
   操作成功/失败有 Toast 提示
   加载状态显示骨架屏或加载动画

2.2 响应式设计
桌面端完整功能
平板端优化触摸拖拽
移动端简化视图（可考虑上下布局）

2.3 辅助功能
键盘快捷键支持
Enter：确认创建/编辑
Esc：关闭弹窗
Delete：删除任务（需确认）
屏幕阅读器支持

4. UI审美（针对带界面的题目）
   · 拒绝浏览器默认样式。请使用组件库（Tailwindcss，AntD，Shadcn 等）保证页面整洁、对齐、美观。

5. 测试要求及流程规范
   5.1 测试分层与工具选择

- 单元测试（Jest / Vitest）：专注验证核心工具函数（如排序计算、日期处理）和数据校验逻辑。
- 组件测试（React Testing Library）：验证核心交互组件（如任务卡片、拖拽容器）的渲染与状态切换。
- 集成测试（Supertest）：确保 Server Actions / API 与数据库（Prisma + SQLite）的读写、级联删除等联动正常。
- E2E 测试（Playwright / Cypress）：模拟真实用户跑通核心业务流（如：创建看板 -> 添加列 -> 创建任务 -> 拖拽排序 -> 编辑保存）。

  5.2 提测与验收标准

- 核心功能必须有对应的自动化测试用例覆盖。
- 所有用例需在本地环境及 CI（如 GitHub Actions）中 100% 绿灯通过，方可合入主分支。

6. DevOps/工程化
   6.1 代码规范与检查

- 统一格式：配置 Prettier 自动格式化代码。
- 静态检查：使用 ESLint 配合 TypeScript 严格模式拦截潜在错误。
- 提交拦截：集成 Husky + lint-staged，在 `git commit` 前自动执行代码格式化与 Lint 检查，确保入库代码质量。

  6.2 版本控制规范

- 分支策略：采用清晰的特性分支工作流（如从 `main` 检出 `feature/xxx` 进行开发，通过 PR 合并）。
- 提交信息：遵循 Conventional Commits 规范（如 `feat:`, `fix:`, `refactor:`, `docs:`），保持 Git 历史清晰并支持自动生成 Changelog。

  6.3 持续集成 (CI)

- 流水线工具：推荐使用 GitHub Actions 配置自动化工作流。
- 自动化卡点：每次向主分支提交 Pull Request 时，CI 必须依次通过：依赖安装 -> 代码静态检查 (Lint) -> 自动化测试 (Test) -> 项目构建 (Build)。

  6.4 部署与发布 (CD)

- 容器化支持：提供基础的 `Dockerfile`，支持将 Next.js 生产构建与 SQLite 数据库打包为一个独立的容器镜像，便于在任何 VPS 服务器上部署。
- Serverless 部署提示：项目默认使用本地 SQLite 数据库；若需部署至 Vercel 等 Serverless 平台，建议将数据库无缝迁移至 Turso (SQLite on edge) 或 PostgreSQL，以解决无状态环境下的数据库写入持久化问题。
